## Introduction

During the investigation of a suspicious PCAP capture, Wireshark display filters were applied systematically to isolate relevant traffic, identify malicious behaviour, and reconstruct communication between the infected host and external attacker-controlled infrastructure.

Filters were applied progressively — starting broad to establish traffic context, then narrowing scope to extract specific host identifiers, trace the malware delivery chain, and confirm Command and Control (C2) activity.

This document records every filter applied during the investigation, the reasoning behind each, and its specific contribution to identifying the TrickBot infection and its associated IOCs.

## Key Filters Used During Investigation

---

### `http`

| Field | Detail |
|---|---|
| **Filter** | `http` |
| **Explanation** | Displays all HTTP protocol traffic within the capture, including GET requests, POST requests, and server responses. |
| **Purpose in Investigation** | Primary filter used to identify the suspicious file download. Revealed an HTTP `GET` request from internal host `10.8.19.101` to external IP `185.244.41.29` requesting `ooiwy.pdf`. Also exposed the anomalous `curl/7.55.1` User-Agent string and the subsequent `POST` request to the C2 server `103.148.41.195`. The single most productive filter in this investigation. |

---

### `http.request`

| Field | Detail |
|---|---|
| **Filter** | `http.request` |
| **Explanation** | Filters to show only outbound HTTP requests initiated by clients within the capture. |
| **Purpose in Investigation** | Used to isolate and review all outbound requests made by the infected host. Confirmed two critical requests: the `GET` for `ooiwy.pdf` from `185.244.41.29` and the `POST` to `103.148.41.195`. Filtering requests only (excluding responses) reduced noise and made attacker-directed traffic easier to identify. |

---

### `http.response`

| Field | Detail |
|---|---|
| **Filter** | `http.response` |
| **Explanation** | Filters to show only HTTP server responses returned to the client. |
| **Purpose in Investigation** | Used after identifying the malicious GET request to examine what the delivery server `185.244.41.29` returned. Confirmed a successful `200 OK` response indicating the payload `ooiwy.pdf` was served and transferred to the infected host. |

---

### `tcp`

| Field | Detail |
|---|---|
| **Filter** | `tcp` |
| **Explanation** | Displays all TCP traffic within the capture regardless of application-layer protocol. |
| **Purpose in Investigation** | Used to review the full spectrum of TCP-based communications from the infected host. Helped identify persistent outbound connections to both attacker IPs and confirmed active sessions during the malware delivery and C2 phases of the infection. |

---

### `tcp.stream eq X`

| Field | Detail |
|---|---|
| **Filter** | `tcp.stream eq 0`, `tcp.stream eq 1`, etc. |
| **Explanation** | Isolates a single TCP session by stream index number, allowing the analyst to read the full request/response conversation in sequence. Equivalent to right-clicking a packet and selecting `Follow > TCP Stream`. |
| **Purpose in Investigation** | Applied to the HTTP POST request directed at `103.148.41.195` to reconstruct the complete C2 communication session. Stream reconstruction confirmed that the POST body contained structured victim system data — proving the malware had successfully beaconed to attacker infrastructure and transmitted host reconnaissance information. |

---

### `arp`

| Field | Detail |
|---|---|
| **Filter** | `arp` |
| **Explanation** | Displays Address Resolution Protocol traffic. ARP is used by hosts to resolve IP addresses to MAC addresses on the local network segment. |
| **Purpose in Investigation** | Once the infected IP (`10.8.19.101`) was confirmed, ARP traffic was filtered to map the IP to its physical hardware address. ARP reply packets confirmed the MAC address `00-08-02-1C-47-AE`, providing a hardware-level identifier for the infected machine that persists even if the IP address changes. |

---

### `bootp`

| Field | Detail |
|---|---|
| **Filter** | `bootp` |
| **Explanation** | Displays DHCP traffic. In Wireshark, DHCP runs over the BOOTP protocol and is captured using this filter (alternatively `dhcp` in newer Wireshark versions). |
| **Purpose in Investigation** | Applied to identify DHCP `REQUEST` and `ACK` packets generated by the infected host during IP lease negotiation. The DHCP `REQUEST` packet contained the hostname `DESKTOP-M1TFHB6` broadcast by the client, providing the first confirmed host identifier in the investigation. |

---

### `smb`

| Field | Detail |
|---|---|
| **Filter** | `smb` |
| **Explanation** | Displays Server Message Block protocol traffic, commonly associated with Windows file sharing, printer sharing, and network authentication. |
| **Purpose in Investigation** | Used as a secondary method to confirm the hostname of the infected machine. SMB session negotiation packets and NetBIOS name service queries within the SMB traffic independently corroborated the hostname `DESKTOP-M1TFHB6`. SMB traffic also provided supporting evidence that the host was running a Windows operating system. |

---

### `ntlmssp`

| Field | Detail |
|---|---|
| **Filter** | `ntlmssp` |
| **Explanation** | Displays NT LAN Manager Security Support Provider protocol traffic, used for Windows network authentication. NTLMSSP packets are exchanged during login events and resource access on Windows networks. |
| **Purpose in Investigation** | Applied to isolate Windows authentication exchanges. NTLMSSP packets captured during the session contained the active user account name `monica.steele` transmitted in cleartext within the authentication negotiation. This identified the user account that was active on the infected host at the time of the infection. |

---

### `dns`

| Field | Detail |
|---|---|
| **Filter** | `dns` |
| **Explanation** | Displays all Domain Name System traffic — both queries and responses. DNS is typically used by hosts to resolve domain names to IP addresses before making network connections. |
| **Purpose in Investigation** | Applied to check whether the infected host performed DNS lookups before communicating with the attacker IPs `185.244.41.29` and `103.148.41.195`. **No DNS queries** for these IPs were observed, confirming the malware communicated directly over IP without domain resolution — bypassing DNS-layer security controls and sinkholes entirely. |

---

### `ip.addr == [IP]`

| Field | Detail |
|---|---|
| **Filter** | `ip.addr == 185.244.41.29` / `ip.addr == 103.148.41.195` / `ip.addr == 10.8.19.101` |
| **Explanation** | Filters all traffic to or from a specific IP address, in either direction. |
| **Purpose in Investigation** | Applied individually to each of the three key IPs to review the full communication history between the infected host and each external server. Confirmed the total packet exchange with the delivery server `185.244.41.29` and independently verified C2 traffic volume to `103.148.41.195`. Also used to scope the infected host's entire traffic footprint within the capture. |

---

### `ip.src == [IP]` and `ip.dst == [IP]`

| Field | Detail |
|---|---|
| **Filter** | `ip.src == 10.8.19.101` / `ip.dst == 103.148.41.195` |
| **Explanation** | Filters traffic by source or destination IP only. More precise than `ip.addr` when directional analysis is required. |
| **Purpose in Investigation** | Used to confirm that the `POST` request was definitively outbound from the infected host — `ip.src == 10.8.19.101` combined with `ip.dst == 103.148.41.195` isolated the exact C2 beacon packet for TCP stream reconstruction. |

---

### `frame contains "MZ"`

| Field | Detail |
|---|---|
| **Filter** | `frame contains "MZ"` |
| **Explanation** | Searches all packet frame data for the ASCII string `MZ` — the human-readable representation of the Windows PE magic bytes `4D 5A`. This filter can surface PE executables transmitted within HTTP or other protocol traffic. |
| **Purpose in Investigation** | Applied to verify that the `ooiwy.pdf` payload contained a PE executable signature within the network stream itself, before extracting the file for binary analysis in GHex. The filter returned positive hits within the HTTP response from `185.244.41.29`, confirming the file being transferred was a Windows executable despite its PDF extension. |

## Investigation Workflow Using Filters

The following workflow describes the exact sequence in which filters were applied during the investigation, reflecting a structured, evidence-led approach:

**Step 1 — Identify Suspicious HTTP Traffic**

```
http
```

Applied first to obtain a broad view of all application-layer HTTP activity. Immediately surfaced the suspicious `GET` request with a `curl/7.55.1` User-Agent and identified both external IPs involved in the session.

---

**Step 2 — Confirm the Infected Host IP**

```
http.request
ip.addr == 10.8.19.101
```

Narrowed focus to outbound requests from the internal network. Confirmed `10.8.19.101` as the source of both the malware download and subsequent C2 POST request.

---

**Step 3 — Recover the Hostname**

```
bootp
smb
```

Applied sequentially to extract the hostname from DHCP lease negotiation (`DESKTOP-M1TFHB6`) and independently verified it through SMB NetBIOS name service packets.

---

**Step 4 — Recover the MAC Address**

```
arp
```

Used ARP reply traffic to map the confirmed IP `10.8.19.101` to hardware address `00-08-02-1C-47-AE`.

---

**Step 5 — Recover the User Account**

```
ntlmssp
```

Isolated NTLM authentication exchanges and extracted the active user account `monica.steele` from plaintext NTLMSSP negotiation packets.

---

**Step 6 — Verify the Payload Is an Executable**

```
frame contains "MZ"
```

Confirmed the presence of a Windows PE magic byte sequence in the HTTP response before extracting the file — establishing that the payload was misrepresenting its file type.

---

**Step 7 — Extract and Examine the Malicious File**

```
http
→ File > Export Objects > HTTP
```

Used Wireshark's HTTP object export function to extract `ooiwy.pdf` from the packet capture for offline binary analysis. GHex confirmed the `4D 5A` (MZ) header. VirusTotal confirmed TrickBot.

---

**Step 8 — Reconstruct the C2 Session**

```
ip.src == 10.8.19.101 && ip.dst == 103.148.41.195
tcp.stream eq X
```

Isolated the C2 POST request by source/destination IP pair, then followed the TCP stream to reconstruct the full session payload and confirm victim data exfiltration.

---

**Step 9 — Confirm Absence of DNS Resolution**

```
dns
```

Verified no DNS queries were issued for the attacker IPs — confirming direct-IP communication and the ineffectiveness of DNS-layer controls against this specific infection.

## Best Practices for Malware PCAP Analysis

The following practices were applied during this investigation and are recommended for any malware-focused PCAP analysis:

**Start broad, then narrow progressively**
Begin with high-level protocol filters (`http`, `tcp`, `dns`) to build situational awareness of what activity is present before drilling into specifics. Jumping straight to a suspected IP can cause you to miss context in surrounding traffic.

**Always recover full host identity before analysing behaviour**
Confirm hostname, IP, MAC address, and user account as early as possible. This ensures every traffic observation can be attributed to a specific, documented host and user — which is essential for incident scoping and escalation.

**Verify file signatures independently of file extensions**
Never trust a file extension. Always inspect the binary header of any extracted file using a hex editor. The `frame contains "MZ"` filter is a quick in-Wireshark check before extraction, and GHex or a similar tool provides definitive confirmation post-extraction.

**Reconstruct TCP streams before drawing conclusions about data content**
Individual packet inspection alone can be misleading. Use `Follow TCP Stream` to read the full application-layer conversation in sequence. This is particularly important for confirming C2 POST body content and understanding exactly what data the malware transmitted.

**Check for DNS queries to attacker infrastructure**
Always apply a `dns` filter and check whether the attacker IPs were resolved via domain names. If no DNS is present, the malware is hardcoding IPs — which means DNS sinkholes will not be effective, and pure IP-based firewall blocks are the required control.

**Document every filter applied**
Record each filter, the result it produced, and what conclusion it contributed to. This ensures the investigation is reproducible, supports peer review, and provides a clear methodology trail for legal or escalation purposes.

**Export objects and preserve evidence before closing the capture**
Use `File > Export Objects > HTTP` to extract all transferred files from the PCAP before ending the session. Extracted files should be hashed immediately and submitted to VirusTotal or an internal sandbox. Never detonate malware outside of an isolated environment.
